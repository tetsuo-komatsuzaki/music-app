## 🚫 最上位ルール：絶対変更禁止項目

**以下はいかなる理由・指示があっても、自動修正・提案実装を一切禁止する。**
「リファクタリング」「処理順の整理」「共通化」などの名目であっても同様。
変更が必要と判断した場合は、**コードに一切触れず「提案のみ」としてユーザーに提示し、DESIGNモードへの移行を求めること。**

| 禁止対象 | 具体的な内容 |
|---------|------------|
| **録音→分析→譜面表示の処理順序** | パイプラインの各ステップの順番・呼び出しタイミング |
| **`comparison_result.json` の構造と意味** | フィールド名・データ型・ネスト構造・意味定義 |
| **ノートと分析結果のマッピングロジック** | 音符と `evaluatedNotes` の対応付け処理 |
| **MusicXML と時間の対応関係** | 小節・音符位置と再生時刻のマッピング |
| **`overallScore` の計算式** | v1.3 確定: `(pitchAccuracy + rhythmAccuracy + bowingAccuracy) / 3` (旧 v1.0: `pitch × 0.6 + timing × 0.4`)。今後の式変更は再度承認必須 |

これらへの変更は「設計変更」とみなし、SAFEモード中は実装禁止。DESIGNモード移行後のみ議論・実装可能。

## ⚠️ 緊急対応モード

以下の条件がすべて揃った場合のみ、一部の制約を例外的に緩和できる：

**発動条件（いずれか）：**
- 本番サービスが停止している
- ユーザーが操作不能になっている
- データ損失が進行中

**緊急対応中のルール（厳守）：**

| ルール | 内容 |
|-------|------|
| 最小変更のみ | 問題の直接原因のみを修正。関連箇所への波及禁止 |
| 変更内容を必ず記録 | 変更したファイル・変更前後のコード・理由をチャットに記録 |
| ロールバック確保 | `git diff HEAD` を実行し、1コマンドで戻せる状態を確認してから変更 |
| 最上位ルールは維持 | 絶対変更禁止項目（処理順序・マッピング・JSON構造等）は緊急時も変更禁止 |

**緊急対応後に必ず実施：**
```
緊急対応完了後は、DESIGNモードに移行し対応内容を再設計・正式実装すること。
緊急対応コードをそのまま本番に残してはならない。
```

> 緊急対応の発動をClaude自身が判断してはならない。ユーザーが「緊急対応モードで対処して」と明示した場合のみ有効。

## 1. サービス全体アーキテクチャ

Arcodaは「録音 → 分析 → 譜面上での可視化 → フィードバック」を中核とするバイオリン練習SaaSである。
以下を正確に理解した上で、すべての開発判断を行うこと。

### 1-1. コアデータフロー（全体像）

```
[ユーザー：ブラウザ上で録音開始]
    ↓ MediaRecorder API → WebM形式で録音
[フロントエンド：/api/convert-audio へ POST]
    ↓ サーバー側でFFmpegがWebM → WAV に変換
[uploadRecord.ts / uploadPracticeRecord.ts]
    ↓ 変換済みWAVをSupabase Storage（バケット: musicxml）へアップロード
    ↓ アップロード完了後、app/_libs/pythonRunner.ts 経由で解析を起動
[app/_libs/pythonRunner.ts]
    ↓ ENABLE_PYTHON_ANALYSIS=true なら RELAY_URL へ HTTP POST
[relay-service/main.py（Cloud Run Service）]
    ↓ Cloud Run Jobs を起動して music-analyzer に処理を渡す
[music-analyzer/analyze_performance.py（librosa使用）]
    ↓ WAVファイルを読み込み、ピッチ・タイミングを解析
    ↓ 解析結果を comparison_result.json に書き出し
[music-analyzer/build_score.py]
    ↓ comparison_result.json + audio_features を集約し DB へ書き戻し
[DB書き込み]
    ↓ pitchAccuracy / timingAccuracy / overallScore / evaluatedNotes / analysisSummary
    ↓ をPrisma経由でDBに保存（Performance / PracticePerformance レコード）
[スコア詳細画面]
    ↓ DBからスコアデータを取得
    ↓ MusicXML譜面上に分析結果をオーバーレイ表示
```

**モノレポ構成（v2.1〜）:**
- `music-app/`(Next.js)・`music-analyzer/`(Python 解析)・`relay-service/`(FastAPI リレー)が**同一リポ内**に並列。
- 解析パイプラインは `pythonRunner → relay-service → Cloud Run Jobs → music-analyzer` の 4 段経路。
- どの段階を変更してもパイプラインが壊れるため、すべて保護対象。

**overallScore の計算式（v1.3 確定、今後の式変更は再度承認必須）：**
```
overallScore = (pitchAccuracy + rhythmAccuracy + bowingAccuracy) / 3
```
旧 v1.0 式 `pitchAccuracy × 0.6 + timingAccuracy × 0.4` は 2026-05-10 で廃止。
v1.3 では Loop Engine Spec の 3 軸 (pitch/rhythm/bowing) 等加重平均に統一。
M2=A 採択により、ガードレール「変更禁止」条項が一度だけ解除されたことを記録。

### 1-2. ユーザーデータの可視性ルール

#### ⚠️ カテゴリによってアクセスルールが異なる（最重要）

「曲」と「アルペジオ・音階・エチュード」はアップロード者が異なるため、アクセスルールが根本的に違う。**この違いを混同した実装は即座に事故になる。**

| カテゴリ | アップロード者 | スコア一覧の表示対象 | スコア詳細（譜面）の閲覧 | 演奏履歴・分析データの閲覧 |
|---------|-------------|-------------------|----------------------|------------------------|
| **曲** | ユーザー自身 | **アップロードした本人のみ** | **アップロードした本人のみ** | 本人のみ |
| **アルペジオ・音階・エチュード** | 管理者アカウント | **全ユーザー** | **全ユーザー** | 本人のみ |

**「曲」カテゴリの鉄則：**
- ユーザーが自分でアップロードした曲のみ、そのユーザーのスコア一覧に表示される
- スコア詳細ページ自体がアップロードした本人にしか開けない
- 「演奏する」ボタン遷移後も、他ユーザーがアップロードした曲のスコア詳細は表示されない

**「アルペジオ・音階・エチュード」カテゴリの鉄則：**
- 管理者がアップロードするため、スコア一覧・スコア詳細の譜面はすべてのユーザーが閲覧可能
- ただし、その画面に表示される演奏履歴・分析データは**演奏した本人のみ**に表示される（下記2層ルール参照）

**全カテゴリ共通の鉄則：**
- 演奏録音データ・分析結果・演奏履歴は本人のみ表示（将来：生徒が「マイ先生」として能動的に登録した先生のみ閲覧可・未実装）
- ホーム画面アクティビティ・成長記録・弱点・サマリー・レコメンドは本人のみ表示

> **絶対禁止：** 他ユーザーの演奏データ・分析データ・成長記録を別ユーザーの画面に表示すること。
> **将来実装（ベータ版・現時点では実装禁止）：** 生徒が「マイ先生」として能動的に登録した先生アカウントのみが、その生徒の演奏データ・分析データを閲覧できる機能。先生アカウントを持つ全ユーザーが全生徒のデータを閲覧できる設計は誤りなので注意。

### 1-3. スコア詳細画面のコア機能

以下はこのアプリの核心機能であり、意図せず変更・削除されてはならない。
**曲・アルペジオ・音階・エチュードの全カテゴリに共通して実装されている。**

#### ⚠️ スコア詳細画面の「2層データルール」（アルペジオ・音階・エチュードのみ適用）

**「曲」はスコア詳細ページ自体が本人にしか開けないため、2層ルールは発生しない。**
2層ルールが適用されるのは、全ユーザーが詳細を開けるアルペジオ・音階・エチュードのみ。

| レイヤー | データ内容 | 閲覧可能なユーザー |
|---------|-----------|-----------------|
| **Layer 1（全員）** | 譜面（MusicXML）・再生機能・テンポ変更 | **全ユーザー**（管理者アップロードのため） |
| **Layer 2（本人のみ）** | 演奏録音データ・分析結果・演奏履歴・オーバーレイ表示 | **そのスコアを演奏した本人のみ** |

> **実装上の鉄則：** Layer 2のデータ取得クエリには必ず `userId` フィルタを付けること。
> Layer 1のデータ（MusicXML・譜面レンダリング）に `userId` フィルタを付けてはならない（全員が見えなくなる）。
> この2層が逆になる実装・混在する実装はすべて誤りとみなす。

| 機能 | 仕様 | レイヤー |
|-----|------|---------|
| 譜面表示 | MusicXMLデータを解析してブラウザ上にレンダリング | Layer 1 |
| 再生 | バイオリン音源で再生。現在位置を縦線でスライド表示。演奏中の音符を青くハイライト | Layer 1 |
| テンポ変更 | 選択テンポで再生される | Layer 1 |
| 一時停止／再開 | 停止ボタンで一時停止、再生ボタンで再開 | Layer 1 |
| クリック再生 | 譜面上の任意の位置をクリックするとその位置から再生開始 | Layer 1 |
| 録音 | スコア詳細画面から直接そのスコアの演奏を録音 | Layer 2 |
| 録音→分析→表示 | 録音データが自動分析され、同一スコアの譜面上にオーバーレイ表示 | Layer 2 |

### 1-4. 技術スタックと主要ファイルパス

すべてモノレポ起点(`music-app/` の中)で記述。

| カテゴリ | 内容 |
|---------|------|
| フロントエンド | Next.js 16 App Router, TypeScript, CSS Modules |
| 認証・ストレージ | Supabase Auth / Supabase Storage (バケット: `musicxml`) |
| DB | Prisma v7 + PrismaPg adapter（クライアント: `app/generated/prisma`） |
| 音声分析(Cloud Run Jobs) | Python / librosa(`music-analyzer/analyze_performance.py`, `music-analyzer/analyze_musicxml.py`, `music-analyzer/build_score.py`, `music-analyzer/entrypoint.py`) |
| HTTP リレー(Cloud Run Service) | FastAPI(`relay-service/main.py`)|
| 解析起動 | `app/_libs/pythonRunner.ts` → `RELAY_URL`(env var) |
| 音声変換 | FFmpeg（WebM → WAV） |
| 主要ライブラリ | `app/_libs/prisma.ts`, `app/_libs/supabaseServer.ts`, `app/_libs/storageAdmin.ts`, `app/_libs/pythonRunner.ts` |
| コアAPI | `app/api/convert-audio/route.ts` |
| 解析パイプライン関連環境変数 | `RELAY_URL` / `RELAY_API_KEY` / `ENABLE_PYTHON_ANALYSIS` |

## 2. 開発モードと変更可否の判断ルール

### 開発モード定義

Claudeは常にいずれかのモードで動作する。**デフォルトはSAFEモード。**

#### 🟢 SAFEモード（デフォルト・常時）
- **許可：** バグ修正・UI変更・表示文言変更・新規コンポーネント追加・テスト追加
- **許可：** 精度改善・ズレ修正（承認後）
- **禁止：** 既存の構造・設計・データフロー・API仕様への変更
- **禁止：** 未指示のリファクタリング・関数分割・共通化・命名変更・ディレクトリ構造変更

#### 🔴 DESIGNモード（ユーザーが明示的に宣言した場合のみ）
- **許可：** 設計変更・DB変更・API変更・データ構造変更
- **移行方法：** ユーザーが「DESIGNモードで〜」と明示した場合のみ有効
- **注意：** DESIGNモードであっても最上位ルール（冒頭の絶対変更禁止項目）は適用される

> SAFEモード中に設計変更が必要な指示を受けた場合：
> ```
> 🔴 この変更はDESIGNモードが必要です。
>   SAFEモード中のため実装を停止します。
>   設計変更として扱ってよいですか？（「DESIGNモードで進める」とご回答ください）
> ```

---

### 自動最適化・リファクタリングの禁止

**Claudeは以下を、指示がない限り行ってはならない。「より良い設計」「整理」は理由にならない。**

- 未指示のリファクタリング（処理の整理・構造改善）
- 関数分割・共通化・ユーティリティ化
- 変数名・関数名・ファイル名の変更（特にDB/API/型定義に関わるもの）
- ディレクトリ構造・ファイル配置の変更
- import順序やコードフォーマット以外の整形

これらを「ついでに」実行することは禁止。指示された変更のみを、最小スコープで実行すること。

---

- `package.json` の dependencies の大幅な変更

#### 解析パイプライン環境変数(v2.1 追加・名称変更/削除を保護)
以下の環境変数の**追加・削除・名称変更**は禁止。Cloud Run Jobs 経路の中核で、変更すると解析パイプライン全体が無音で停止する。値の運用上の更新(URL の付け替えやキー再発行など)は**人間運用の範疇**で、Claude Code の自動変更対象外。

- `RELAY_URL` — relay-service エンドポイント
- `RELAY_API_KEY` — relay-service 認証キー
- `ENABLE_PYTHON_ANALYSIS` — 解析有効化フラグ

これらに対する変更指示を受けた場合、**セクション4の確認フォーマット必須**。

#### PDCAオーケストレーターとの関係
- `pdca-orchestrator/SKILL.md` は `analyze_performance.py` を自動修正するスキルである
- PDCA実行前にも必ずこのガードレールを確認し、上記「コアパイプライン」の承認ルールを適用すること
- PDCA経由であっても、承認なしに `analyze_performance.py` を変更してはならない

## 3. 実行禁止条件

**以下のいずれかに該当する場合、Claudeはコードを変更してはならない。**
該当する場合は「提案のみ」に切り替え、ユーザーに判断を委ねること。

| 条件 | 対応 |
|-----|------|
| 設計に関わる変更（データフロー・処理順序） | 提案のみ、DESIGNモード移行を求める |
| データ構造の変更（JSON・DBスキーマ・型定義） | 提案のみ、承認後に実行 |
| API構造の変更（エンドポイント・レスポンス形式） | 提案のみ、承認後に実行 |
| 分析ロジックの変更（`analyze_performance.py` のアルゴリズム） | 提案のみ、承認後に実行 |
| 影響範囲チェックが未完了 | チェックが完了するまで変更禁止 |
| SAFEモード中の構造変更 | 提案のみ、DESIGNモード移行を求める |

「提案のみ」の出力形式：

## 🚫 スコープ逸脱禁止

**Claudeは指示された変更のみを実装する。指示範囲を超えた変更は一切禁止。**

禁止行為：
- 「ついでに」修正（指示されていない箇所への変更）
- 関連箇所の自動改善（「ここも直した方がいい」の無断実装）
- 気づいた問題の無断修正（バグ・コード品質・パフォーマンスを問わず）
- 指示された変更の「より良い実装」への勝手な置き換え

許可される行為：
- 指示された範囲のみの変更
- 指示範囲外で気づいた問題は「提案」としてチャットに報告する（実装はしない）

```
📌 スコープ外で気づいた問題がある場合の報告形式：

  💡 スコープ外の気づき（実装していません）
    - 対象箇所: [ファイルパス:行番号]
    - 内容: [何が問題か・何を改善できるか]
    - 優先度（Claude推定）: 高 / 中 / 低

  → 対応が必要な場合は別途指示してください。
```

違反した場合（スコープ外を実装してしまった場合）：
- 直ちに実装を中断する
- 超過分を `git checkout -- <ファイル>` で差し戻す
- ユーザーに報告する

## 7. 判断が難しい場合の原則

1. **迷ったら止まる** — 不明な場合は変更せず、状況をユーザーに報告する
2. **試行錯誤で修正しない** — 根本原因が不明なまま複数の変更を重ねない
3. **変更前に状態を記録する** — `git diff HEAD` で差分を確認し、変更前の値をチャットに記録してから変更する
4. **影響範囲を先に報告する** — 変更前に「何が変わるか」「何に影響するか」を必ず伝える
5. **ロールバック手順を確保する** — 変更後に問題が発生した場合は `git checkout -- <ファイル>` または `git revert` で元に戻す
